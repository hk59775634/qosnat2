# 发布规则

本文档为 **跨项目通用** 的发布流程与约定。首个落地项目为 [qosnat2](README.md)；[qosnat-vpp](/opt/qosnat-vpp) 等新项目应 **借鉴本规则** 并替换产品名、构建产物与 manifest 路径。

项目专属细节见各仓库 `release/RELEASE.md`（若有）。

---

## 1. 目标与原则

| 原则 | 说明 |
|------|------|
| **可复现** | 发布由 CI 执行固定脚本，避免「本机能编、线上不能编」 |
| **可回滚** | 版本号与 Git tag 一一对应；保留版本清单与归档说明 |
| **人读优先** | Release 正文来自人工编写的更新说明，**不用** commit 列表代替 |
| **先说明后发布** | 未通过 `release-notes validate` 不得触发正式发版 |
| **生产用 Release** | 除开发环境外，目标机 **统一安装 Release 包**，不在生产机源码编译 |
| **发版与文档同提交** | `pending-release-notes.md` 与功能代码 **同一 commit** 进入 `main` |

---

## 2. 版本号规范

### 2.1 格式（默认，所有网关类产品沿用）

```
YYYYMMDD + NN
```

| 部分 | 含义 |
|------|------|
| `YYYYMMDD` | **UTC** 日期，8 位 |
| `NN` | 当日内序号 `01`–`99`，按 manifest 中当日已有版本递增 |

**示例**：`2026061102` → Git tag **`v2026061102`**

### 2.2 版本清单（manifest）

- 路径约定：`releases/<产品名>-versions.json`
- 示例：[releases/qosnat2-versions.json](releases/qosnat2-versions.json)
- 字段：`id`、`tag`、`published_at`、`summary`（来自更新说明「概要」一句）
- 默认仅保留最新 **5** 条（`max_keep`，可用环境变量 `RELEASE_MAX_KEEP` 覆盖）
- 分配版本号：**仅 CI 或** `./scripts/release-version.sh <产品> next`；禁止手改 tag 造版本

### 2.3 新项目适配

1. 复制 `scripts/release-version.sh`，将 `qosnat2` 改为新产品名（或参数化 `product`）
2. 新建 `releases/<产品>-versions.json`：`{"schema":1,"max_keep":5,"versions":[]}`
3. GitHub Release tag 统一为 **`v` + 10 位版本号**

## 3. 更新说明（Release Notes）

### 3.1 文件位置

| 文件 | 用途 |
|------|------|
| `release/pending-release-notes.template.md` | 模板（勿删改结构） |
| `release/pending-release-notes.md` | **待发版**说明（发版前编辑，成功后 CI 重置为模板） |
| `releases/notes/<版本号>.md` | **已归档**说明（只增不改） |

### 3.2 栏目（必填结构）

```markdown
## 概要        ← 一句话，写入 manifest summary，禁止占位符
## 新增
## 优化
## 修复
## 删除
## 其他
```

- **概要**：必须有一条有效文字（非「（一句话概括…）」等模板句）
- **新增 / 优化 / 修复 / 删除 / 其他**：至少 **一栏** 有一条有效说明（非「（无）」）
- 不得保留 **`## 待分类`** 及以下内容（`draft` 生成的待分类节在提交前须归类并删除）

### 3.3 命令

```bash
./scripts/release-notes.sh draft      # 可选：自上一 tag 列出 commit 草稿
# 编辑 release/pending-release-notes.md
./scripts/release-notes.sh validate   # 发版前本地/CI 必过
./scripts/release-notes.sh finalize <id>   # 一般由 CI 调用
./scripts/release-notes.sh reset        # 手动重置 pending 为模板
```

## 4. 发布前检查清单

### 4.1 开发自测

- [ ] 单元测试 / 构建通过（如 `go test ./...`、`npm run build`）
- [ ] 关键路径冒烟（项目自有 `scripts/acceptance-*.sh` 或 API 冒烟）
- [ ] 无已知 P0 未修复项（若已知，写在「其他」并评估是否延后发版）

### 4.2 更新说明

- [ ] 已运行 `draft`（可选）并完成人工归类
- [ ] `validate` 通过
- [ ] `pending-release-notes.md` 已与功能变更 **同一 commit** 提交

### 4.3 构建机要求（CI 与本地 `build-release.sh` 一致）

- 平台：**Ubuntu 24.04 x86_64**（交叉编译需单独验收）
- 工具链：见各项目 `scripts/build-release.sh` 与 `.github/workflows/release.yml`

### 4.4 Git 约定（发版相关）

- **禁止** 在未请求时自行 `git commit` / `push`（自动化 Agent 须等用户确认发版）
- 发版产生的 bot commit 信息格式：`chore(release): <产品名> <版本号>`
- **禁止** `commit --amend` 已推送的 release commit（除非用户明确要求且符合 amend 条件）
- **禁止** 在 release commit 中夹带未说明的功能变更

## 5. 触发发布（CI）

### 5.1 自动触发（push `main`）

默认 workflow（qosnat2：[.github/workflows/release.yml](.github/workflows/release.yml)）在 push `main` 时运行，但 **忽略** 仅下列路径的变更：

- `releases/**`
- `docs/**`
- `**/*.md`
- `web/**`（仅前端时）
- `.github/**`

即：**后端 / 脚本 / 数据面** 变更 push `main` → 自动发版（且 pending 说明已校验）。

### 5.2 手动触发

| 场景 | 方式 |
|------|------|
| 仅前端 / 仅文档但仍需发版 | `gh workflow run release.yml` |
| 重传同版本二进制（**不新造版本号**） | `gh workflow run republish-release.yml -f version_tag=vXXXXXXXXXX` 或 `./push.sh <tag>` |

### 5.3 CI 步骤（标准顺序）

1. 校验 `release/pending-release-notes.md`
2. `release-version.sh <产品> next` 分配版本号
3. `build-release.sh` 构建产物
4. 用 pending 说明生成 **GitHub Release 正文**（去掉模板头与「待分类」）
5. `release-notes.sh finalize <id>` → 归档 `releases/notes/<id>.md`，重置 pending
6. 更新 manifest（写入 `summary`）
7. 创建 GitHub Release（tag + 附件）
8. 自动 commit：`releases/*`、`release/pending-release-notes.md`

## 6. Release 产物约定

### 6.1 qosnat2（参考实现）

| 产物 | 说明 |
|------|------|
| `dist/qosnatd-linux-amd64` | 控制面单文件（`-tags release`，内嵌 Web + BPF） |
| `dist/qosnat2-linux-amd64.tar.gz` | 二进制 + `lib/classify.bpf.o` + 可选 `lib/dnsmasq-chnroutes` |

构建：`./scripts/build-release.sh`

目标机：**禁止**依赖源码树 `web/dist`；用 Release 包安装/升级。见 [README.md](README.md) 一键安装与版本切换。

### 6.2 新项目（如 qosnat-vpp）建议

| 项 | 建议 |
|----|------|
| 二进制命名 | `dist/<daemon>-linux-amd64` |
| 压缩包 | `dist/<产品>-linux-amd64.tar.gz` |
| 内嵌资源 | 控制面静态资源、配置模板一并 embed 或 tarball |
| 构建脚本 | `scripts/build-release.sh` 单一入口 |
| 平台 | 首版固定 `linux/amd64`，扩展需单独验收文档 |

## 7. 发布后

### 7.1 查阅渠道

- GitHub **Releases** 页（正文 + 附件）
- 仓库 `releases/notes/<版本>.md`
- Web UI **系统 → 版本管理**（若产品实现版本切换）

### 7.2 生产升级

- 从 GitHub Release 下载对应 tag 资产
- 覆盖安装路径（如 `/usr/local/bin/qosnatd`）
- 重启 systemd 服务
- 验证 `/api/v1/health` 与版本号

### 7.3 热修复已发布版本

**不增加新版本号**，仅替换二进制：

```bash
./push.sh v2026061102
# 或
gh workflow run republish-release.yml -f version_tag=v2026061102
```

适用：构建失败、漏传附件、无功能变更的编译修复。  
**不适用**：有任何用户可见行为变更 → 必须走新发版 + 新更新说明。

### 7.4 发版后沟通

- 概要应与 manifest `summary`、Release 标题一致
- 破坏性变更写在「删除」或「其他」，并注明升级注意

---

## 8. 禁止事项

| 禁止 | 原因 |
|------|------|
| 跳过 `validate` 强行发版 | Release 正文不可读或为空 |
| 用自动生成 commit log 代替人工说明 | 运维与用户无法知悉影响 |
| 生产环境 `go build` / `npm run build` 安装 | 不可复现、难回滚 |
| 同一天手工编造冲突版本号 | 与 manifest 不一致 |
| republish 夹带行为变更却不升版本 | 破坏版本与产物对应关系 |
| 在 release 自动 commit 中混入无关文件 | 审计与回滚困难 |

## 9. 新项目接入清单

复制到新仓库时，建议具备：

```
release/
  pending-release-notes.template.md
  pending-release-notes.md          # 从模板复制
  RELEASE.md                        # 产品专属补充（可选）
releases/
  <产品>-versions.json
  notes/
scripts/
  release-notes.sh                  # 通用，一般无需改产品名
  release-version.sh                # 注册产品名
  build-release.sh
.github/workflows/
  release.yml
  republish-release.yml             # 可选
```

**release.yml 中需替换**：

- workflow 名、`concurrency.group`
- `release-version.sh` 的产品参数
- `build-release.sh` 产物路径与 `files:` 列表
- Release 标题前缀（如 `qosnat-vpp`）

**Agent / 开发者**：发版前阅读本文 + 项目 `release/RELEASE.md`；Cursor 规则可引用「发布规则见 `/opt/qosnat2/发布规则.md`」。

## 10. 命令速查

```bash
# 更新说明
./scripts/release-notes.sh draft
./scripts/release-notes.sh validate

# 版本
./scripts/release-version.sh qosnat2 latest
./scripts/release-version.sh qosnat2 next

# 本地构建（不发布）
./scripts/build-release.sh

# 手动发版 / 重传
gh workflow run release.yml
gh workflow run republish-release.yml -f version_tag=v2026061102
./push.sh v2026061102
```
