# 系统优化内存模型说明

## Google 建议是否合理（16GB 示例）

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `nf_conntrack_max` | 16,777,216 | 约 1677 万并发跟踪条目 |
| `nf_conntrack_buckets` | 4,194,304 | 约为 max 的 1/4，且为 2 的幂 |

**结论：在「专用优化内存 = 总内存 50%」前提下合理。**

推算（qosnat2 采用同一模型）：

1. 总内存 16GB → 优化专用 **8GB**（50%）
2. 其中 **75%** 给 conntrack 表 ≈ **6GB**
3. 单条约 **384 字节**（内核常见 300–400B）→ `6GB / 384 ≈ 16,777,216`
4. 哈希桶 `max / 4 = 4,194,304`（2 的幂），与建议一致

注意：

- 需保证 `nf_conntrack` 模块已加载，且内核 `nf_conntrack_max` 上限允许该值
- 实际可达连接数还受 CPU、PPS、路由表、文件句柄（`fs.file-max`）限制
- 内存不足时 OOM 风险上升，**不建议**在 <4GB 机器上使用千万级 max

## qosnat2 计算公式

```
optimization_mb = mem_total_mb × 0.5
conntrack_mem_mb = optimization_mb × 0.75
conntrack_max = conntrack_mem_mb × 1024² / 384
conntrack_buckets = round_up_power2(conntrack_max / 4)
fs.file-max = max(conntrack_max × 2, 524288)
```

档位（low/medium/high）仍影响：RPS、txqueuelen、TCP 超时、perf_preset 等；**conntrack 仅按内存计算**。
